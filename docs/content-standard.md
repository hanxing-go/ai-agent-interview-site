# Content Standard

本项目的核心不是收集尽可能多的题，而是沉淀可复习、可表达、可追问的高质量面试内容。

## 推荐题目结构

每道题建议包含：

```json
{
  "id": "rag-001",
  "topic": "RAG",
  "question": "RAG 如何缓解大模型幻觉？",
  "difficulty": "medium",
  "tags": ["RAG", "幻觉", "检索增强", "工程实践"],
  "companies": ["字节", "百度"],
  "novice_answer": "适合快速背诵的简短回答。",
  "expert_answer": "结构化、专业、可面试表达的回答。",
  "key_points": ["检索质量", "引用来源", "拒答机制"],
  "pitfalls": ["不要说 RAG 可以彻底消除幻觉"],
  "follow_ups": ["如何评估 RAG？", "RAG 和微调怎么取舍？"],
  "references": [{"title": "来源", "url": "https://..."}]
}
```

## 难度定义

| 难度 | 含义 |
|------|------|
| `basic` | 基础概念题，适合入门背诵 |
| `medium` | 高频面试题，需要结构化回答 |
| `hard` | 高阶追问，需要工程经验 |
| `system-design` | 系统设计题，需要整体方案 |
| `project` | 项目深挖题，需要结合个人项目 |

## 回答标准

好的答案应该：

1. 能直接口头表达。
2. 先给结论，再展开原因。
3. 有工程化细节，而不是只背概念。
4. 能覆盖常见追问。
5. 明确易错点。
6. 不夸大技术效果。

## 示例：好答案结构

```text
结论：RAG 不能彻底消除幻觉，但可以显著降低事实性错误。

展开：
1. 通过检索外部知识降低模型凭空编造的概率；
2. 通过引用来源提高可验证性；
3. 通过重排、过滤和拒答机制控制低质量上下文；
4. 通过评估集持续监控召回率、准确率和回答一致性。

易错点：不要把 RAG 说成万能方案，检索不到、召回错、上下文冲突时仍会幻觉。
```

## 内容合规

- 可以分享自己的复盘和公开链接摘要。
- 不要复制大段付费课程、书籍、私密文档。
- 不要泄露面试官、候选人、内部系统等隐私信息。
- 真实公司面经如不确定是否可公开，请脱敏处理。
